## Why this matters

ObjectCore is "a software factory whose deliverable is the system that produces
plugins." The forge pipeline (grill → plan → scaffold → gate → release → ingest)
is built and unit-tested, **but it has never produced a real end-user plugin** —
all six plugins in `plugins/` are infrastructure (five meta-plugins + one demo),
and `plugin-forge`'s prose skills are described in `CLAUDE.md` as "currently a
skeleton." Until the factory makes something real, we don't know where the prose
or the engine falls short. This spike runs the deterministic pipeline on one
concrete, genuinely-useful plugin to (a) grow the catalog with the first real
entry and (b) produce a **gap report** that tells the maintainer exactly what to
harden in the forge prose before scaling plugin production.

## Current state

- `bun run forge:scaffold <spec.json> [--force]` (`scripts/forge-scaffold.ts`)
  reads a `PluginSpec` JSON, scaffolds the plugin, defaults its `author` to the
  config owner, then runs `syncAndGate` (re-derive + validate + write
  `marketplace.json` + offline output/coverage evals). It prints
  "Next: `bun run eval` to run the activation gate (needs an API key)."
- The scaffolder (`packages/forge/src/scaffold.ts`) enforces the hard rules at
  write time and **refuses to emit a skill without activation cases**
  (`scaffold.ts:84-88`). A `PluginSpec` with a skill MUST include `activation`.
- `PluginSpec` shape (`packages/forge/src/types.ts:18-34`): `name`,
  `description`, optional `version` (defaults `0.0.1`), `author`, `license`
  (defaults `MIT`), `keywords`, `category`, `repository`, `skills[]`,
  `commands[]`, `activation[]`, `expectEntry`.
- Activation case shape — mirror an existing one,
  `plugins/plugin-validator/evals/activation.json`:

```json
{ "prompt": "…", "expect": "<skill-name-or-null>", "note": "…" }
```
  `expect` is the skill `name` that should fire, or `null` for "nothing should
  fire" (a negative case). **Coverage evals require every skill to have at least
  one positive case** (`expect` = that skill's name).
- The offline gate (`syncAndGate` in `scripts/_finalize.ts`) runs output +
  coverage evals and does NOT need a key. The **activation gate** is the separate
  `bun run eval`, whose `AnthropicJudge` needs `ANTHROPIC_API_KEY`; without a key
  the activation layer is reported as **skipped** (never silently passed).

**Hard rule that governs the finish line** (`AGENTS.md`): *No plugin enters the
catalog without passing validation AND its activation eval.* So this plugin is
only truly "shipped" once `bun run eval` is green with a real key. Without a key,
you produce an offline-green plugin and STOP at Step 4 (see that step).

**Repo conventions to match:**
- A skill's frontmatter `description` is its trigger surface — precise and
  distinctive. The default skill body generated by the scaffolder
  (`scaffold.ts:37-44`) already says this; if you hand-write a body, keep it
  tight.
- Negative activation cases should include a **confusability negative** — a
  prompt that looks related but must NOT fire (see plugin-validator's
  "Add validation to the email field…" case).

## Steps

### Step 1: Author the PluginSpec for the vehicle plugin

Create `plans/notes/commit-craft.spec.json` with the spec below. This plugin
helps a user write a clear git commit message for staged changes — a small, real,
single-skill plugin with a trigger surface distinct from all six existing
(meta-)plugins.

> If the operator has a preferred first plugin, you may substitute its idea, but
> keep the exact same structure (one skill, ≥1 positive + ≥2 negative activation
> cases including one confusability negative) and the same step sequence.

```json
{
  "name": "commit-craft",
  "description": "Write clear, conventional git commit messages from your staged changes.",
  "version": "0.1.0",
  "keywords": ["git", "commits", "workflow"],
  "category": "workflow",
  "skills": [
    {
      "name": "writing-commit-messages",
      "description": "Draft a clear, conventional-commits-style git commit message that summarizes the staged diff. Use when the user is about to commit, asks for a commit message, or wants their commit wording improved."
    }
  ],
  "activation": [
    {
      "prompt": "I just staged a bug fix and a small refactor — write me a good commit message for it.",
      "expect": "writing-commit-messages",
      "note": "positive — drafting a commit message from staged changes"
    },
    {
      "prompt": "Rewrite this commit message to follow conventional commits: 'fixed stuff'.",
      "expect": "writing-commit-messages",
      "note": "positive — improving commit wording"
    },
    {
      "prompt": "What's a good recipe for dinner tonight?",
      "expect": null,
      "note": "clear negative"
    },
    {
      "prompt": "Review my plugin against the marketplace rules before I add it to the catalog.",
      "expect": null,
      "note": "confusability negative — about plugin validation, must NOT fire the commit skill (that's plugin-validator's job)"
    }
  ]
}
```

**Verify**: the file is valid JSON — `bun -e "JSON.parse(require('fs').readFileSync('plans/notes/commit-craft.spec.json','utf8')); console.log('ok')"` → prints `ok`.

### Step 2: Scaffold the plugin and run the offline gate

```
bun run forge:scaffold plans/notes/commit-craft.spec.json
```

Expected: it prints `✓ scaffolded commit-craft -> plugins/commit-craft`, lists
the written files (`.claude-plugin/plugin.json`, `skills/writing-commit-messages/SKILL.md`,
`evals/activation.json`, `evals/output.json`), then `✓ catalog re-derived (7 plugins)`
and an output+coverage eval report that ends green.

**While this runs, note for the report (Step 5):** Did the scaffolder need
anything the spec didn't obviously provide? Was the generated `SKILL.md` body
usable as-is, or clearly a stub needing prose? Did the offline gate pass on the
first try?

**Verify**: command exits 0; `git status` shows new files under
`plugins/commit-craft/` and a modified `.claude-plugin/marketplace.json`.

### Step 3: Confirm the deterministic gates are green

```
bunx tsc && bun test && bun run check:catalog
```

Expected: all exit 0. `check:catalog` re-derives and asserts the committed
`marketplace.json` byte-matches — it will, because Step 2 wrote it through the
same `deriveCatalog` seam.

**Verify**: all three exit 0.

### Step 4: Run the activation gate (the real finish line)

```
bun run eval
```

- **If `ANTHROPIC_API_KEY` is set**: the activation layer runs the four
  `commit-craft` cases through the judge against the whole catalog's trigger
  surfaces. Expected: all green. If a `commit-craft` case fails (e.g. the
  confusability negative fires the skill, or a positive doesn't), the skill's
  trigger-surface `description` needs tightening — edit
  `plugins/commit-craft/skills/writing-commit-messages/SKILL.md` frontmatter
  `description` only, re-run `bun run eval`, and record what you changed in the
  report. Do NOT loosen the activation cases to make them pass.
- **If `ANTHROPIC_API_KEY` is NOT set**: the activation layer reports **skipped**.
  Per the hard rule, the plugin is not yet fully gated. **STOP here**: leave the
  plugin and re-derived catalog in place, write the report (Step 5) noting the
  activation gate is pending, and report back that a key is required to finish.
  Do not delete the plugin and do not claim it is shipped.

**Verify**: `bun run eval` shows the activation layer green (key present) or
explicitly skipped (no key) — never silently passed.

### Step 5: Write the gap report (the deliverable)

Create `plans/notes/003-forge-spike-findings.md` capturing what the spike
revealed. Structure it as:

- **Outcome**: did `commit-craft` reach a fully green `bun run check` (with key)
  or stop at the offline gate (no key)? One paragraph.
- **Forge prose gaps**: `plugin-forge`'s `specifying`/`planning`/
  `writing-great-skills` skills are the human/frontier-model guidance that
  produces a `PluginSpec`. Having now hand-authored a spec and run it, list what
  that guidance is missing or vague about (e.g. how many activation cases, how to
  pick a distinctive trigger surface, the confusability-negative requirement,
  category conventions). Cite the skill files you read.
- **Engine/CLI friction**: anything awkward in `forge:scaffold` /
  `scaffoldPlugin` / `syncAndGate` (error messages, defaults, the
  spec-input-is-throwaway ergonomics, the two-step offline-then-activation gate).
- **Recommendations**: 2–4 concrete, scoped follow-ups (each a candidate plan),
  e.g. "flesh out `planning` skill with a spec checklist," "make `forge:scaffold`
  accept the spec on stdin," "add a `forge:new` that runs the activation gate
  inline when a key is present."

**Verify**: the file exists and contains the four sections above.

## STOP conditions

Stop and report back (do not improvise) if:

- `bun run forge:scaffold` errors (e.g. complains the spec is missing activation
  cases or violates a hard rule) and the cause isn't an obvious typo in your spec
  — capture the exact error in the report.
- The offline gate (`check:catalog` or the output/coverage evals) is red after a
  clean scaffold — that would indicate an engine bug; report it rather than
  hand-editing generated files.
- `ANTHROPIC_API_KEY` is unset (stop at Step 4 as described — this is an expected,
  documented stop, not a failure).
- Making an activation case pass would require changing the *cases* rather than
  the skill's `description` — report it; never weaken the eval to go green.
